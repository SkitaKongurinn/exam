# Билет №18 — ACL, WL/BL, REBAC, ABAC, linux rwx, матрица прав

## 0. Как понять этот билет простыми словами

Билет про права доступа.

После входа система решает:

```text
Можно ли пользователю read/create/update/delete ресурс?
```

---

## 1. Теоретический вопрос 1

**Вопрос:** Что такое ACL и списки WL/BL? Приведите пример для ресурса «подписки».

### Простое объяснение

ACL — Access Control List, список прав к ресурсу.

```text
owner: read, update
admin: read, create, update, delete
guest: denied
```

WL — white list, кому разрешено. BL — black list, кому запрещено.

### Готовый ответ на 1 вопрос

ACL, или Access Control List, — список правил доступа к ресурсу. В ACL указывается, какие пользователи или роли могут выполнять операции с конкретным объектом. Например, владелец подписки может читать и отменять её, а admin может читать, создавать, изменять и удалять подписки.

WL — white list, список разрешённых субъектов. BL — black list, список запрещённых субъектов. Например, можно разрешить доступ к подписке владельцу и администраторам через WL, а конкретному заблокированному пользователю запретить доступ через BL.

## 2. Теоретический вопрос 2

**Вопрос:** Сравните REBAC, ABAC и linux rwx. Когда какую модель использовать?

### Простое объяснение

REBAC — доступ по отношениям: пользователь владеет подпиской.

ABAC — доступ по атрибутам: role=admin, status=active, region=RU.

Linux rwx — read/write/execute, простая модель для файлов.

### Готовый ответ на 2 вопрос

REBAC основывает доступ на отношениях между субъектом и объектом. Например, пользователь может читать подписку, если он является её владельцем. Эта модель удобна для CRM, где важны связи между пользователями, клиентами, подписками и менеджерами.

ABAC принимает решение на основе атрибутов пользователя, ресурса и контекста. Например, доступ разрешён, если роль пользователя admin, статус подписки active, а операция выполняется из разрешённой сети. Linux rwx — простая модель прав read, write, execute, которая хорошо подходит для файловой системы, но менее гибкая для сложной CRM.

## 3. Практическое задание

**Задание:** Составьте матрицу прав для ролей admin и subscriber.

### Решение

| Роль | Ресурс | Read | Create | Update | Delete |
|---|---|---:|---:|---:|---:|
| admin | User | ✅ | ✅ | ✅ | ✅ |
| admin | Payment | ✅ | ✅ | ✅ | ✅ |
| admin | Subscription | ✅ | ✅ | ✅ | ✅ |
| subscriber | User | ✅ только себя | ❌ | ✅ только себя | ❌ |
| subscriber | Payment | ✅ только свои | ❌ | ❌ | ❌ |
| subscriber | Subscription | ✅ только свои | ✅ | ✅/cancel только свои | ❌ |

Текстом:

```text
Admin имеет полный доступ.
Subscriber читает и обновляет только свои данные.
Subscriber может создать подписку и отменить свою подписку.
Subscriber не может удалять пользователей, платежи и чужие подписки.
```

---

## 4. Краткая версия для устного пересказа

ACL — список прав доступа. WL разрешает, BL запрещает. REBAC выдаёт права по отношениям, например владелец подписки может её читать. ABAC использует атрибуты: роль, статус, контекст. Linux rwx — read/write/execute. Для CRM часто подходят REBAC и ABAC.

---

## 5. Что обязательно сказать

- ACL = список прав.
- WL = кому разрешено.
- BL = кому запрещено.
- REBAC = права по отношениям.
- ABAC = права по атрибутам.
- rwx = read/write/execute.

---

## 6. Типичные ошибки

1. Перепутать ACL и ABAC.
2. Дать subscriber доступ к чужим платежам.
3. Разрешить subscriber удалять User.
4. Не указать «только свои».
5. Не объяснить REBAC через отношения.
