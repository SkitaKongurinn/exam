# Билет №1 — ФТ, Издатель, Подписчики 1/2/3, pub/sub

## 0. Как понять этот билет простыми словами

Представь Telegram-канал: автор публикует пост, а подписчики получают уведомление. Автор не пишет каждому подписчику лично.

```text
Издатель → публикует событие → система доставляет его Подписчику 1, 2 и 3
```

Это называется **pub/sub**: publish — опубликовать, subscribe — подписаться.

---

## 1. Теоретический вопрос 1

**Вопрос:** Что такое функциональные требования (ФТ) в контексте WEB-приложения подписок? Перечислите ключевые ФТ для роли «Издатель» и опишите его место в архитектуре pub/sub.

### Простое объяснение

**Функциональные требования** — это то, **что система должна делать**.

Примеры ФТ:

```text
Система должна позволять создать подписку.
Система должна позволять Издателю опубликовать событие.
Система должна доставить событие подписчикам.
Система должна сохранить данные.
Система должна записать Log/Audit.
```

**Издатель** — участник системы, который создаёт и публикует событие: «новая подписка», «подписка отменена», «создан платёж».

В pub/sub Издатель не знает всех подписчиков. Он публикует событие, а сервис подписок, Kafka или очередь доставляют его получателям.

### Готовый ответ на 1 вопрос

Функциональные требования — это требования, которые описывают, что должна делать система. В WEB-приложении подписок они определяют основные действия: создание подписки, публикацию события, доставку события подписчикам, сохранение данных и запись логов или аудита.

Роль «Издатель» отвечает за создание и публикацию событий. Например, Издатель может создать событие «новая подписка». В архитектуре pub/sub Издатель не отправляет сообщение каждому подписчику напрямую. Он публикует событие в систему, а сервис подписок или брокер сообщений доставляет событие нужным подписчикам.

## 2. Теоретический вопрос 2

**Вопрос:** Чем отличаются роли «Подписчик 1», «Подписчик 2» и «Подписчик 3» в системе? Какие типы событий или данных может получать каждый из них?

### Простое объяснение

Подписчик 1, 2 и 3 — разные типы получателей событий. Они могут отличаться объёмом данных и назначением.

| Роль | Что получает |
|---|---|
| Подписчик 1 | Базовое уведомление: «создана новая подписка» |
| Подписчик 2 | Расширенные данные: пользователь, тип подписки, статус |
| Подписчик 3 | Служебные данные: аудит, аналитика, платежи, внутренняя обработка |

Важно: подписчики не обязательно люди. Это могут быть роли, сервисы или обработчики внутри системы.

### Готовый ответ на 2 вопрос

Подписчик 1, Подписчик 2 и Подписчик 3 — это разные получатели событий в системе подписок. Они могут отличаться типом получаемых данных и назначением.

Например, Подписчик 1 получает базовое уведомление о событии, такое как «создана новая подписка». Подписчик 2 может получать расширенную информацию: данные пользователя, тип подписки или статус. Подписчик 3 может получать служебные события для аудита, аналитики, расчёта платежей или внутренней обработки.

Таким образом, все три подписчика получают событие от одного Издателя, но каждый может использовать его для своей задачи.

## 3. Практическое задание

**Задание:** Опишите пошаговый сценарий: Издатель публикует событие «новая подписка», событие доставляется трём подписчикам. Укажите участников и направление сообщений.

### Решение

Участники:

```text
Издатель
Web client
Backend gRPC API
Сервис подписок
Kafka / очередь сообщений
Подписчик 1
Подписчик 2
Подписчик 3
Log / Audit
```

Алгоритм:

```text
1. Издатель в Web client создаёт событие "новая подписка".
2. Web client отправляет запрос в backend gRPC API.
3. Backend проверяет данные.
4. Backend создаёт событие NewSubscriptionCreated.
5. Backend передаёт событие в сервис подписок.
6. Сервис подписок определяет получателей.
7. Сервис подписок публикует событие в Kafka или очередь.
8. Kafka/очередь доставляет событие Подписчику 1.
9. Kafka/очередь доставляет событие Подписчику 2.
10. Kafka/очередь доставляет событие Подписчику 3.
11. Система записывает Log/Audit.
```

Схема:

```text
Издатель → Web client → backend gRPC API → сервис подписок → Kafka/очередь
                                                               ├→ Подписчик 1
                                                               ├→ Подписчик 2
                                                               └→ Подписчик 3
```

Готовый текст практики:

Издатель создаёт событие «новая подписка» через Web client. Клиент отправляет запрос в backend gRPC API. Backend проверяет данные и формирует событие NewSubscriptionCreated. Затем backend передаёт событие в сервис подписок.

Сервис подписок определяет список получателей и публикует событие в Kafka или очередь сообщений. После этого событие доставляется Подписчику 1, Подписчику 2 и Подписчику 3. Каждый подписчик получает нужные ему данные: базовое уведомление, расширенную информацию или служебное событие. После обработки система записывает технический лог и при необходимости audit-запись.

---

## 4. Краткая версия для устного пересказа

Функциональные требования — это то, что система должна делать. В приложении подписок Издатель публикует событие, например «новая подписка». В architecture pub/sub Издатель не отправляет событие каждому подписчику напрямую, а передаёт его в систему. Сервис подписок через Kafka или очередь сообщений доставляет событие Подписчику 1, Подписчику 2 и Подписчику 3. Подписчики отличаются типом получаемых данных: базовое уведомление, расширенные данные или служебная информация для аудита, аналитики или платежей.

---

## 5. Что обязательно сказать

- ФТ описывают, что система должна делать.
- Издатель публикует событие, но не знает конкретных подписчиков.
- Pub/sub разделяет отправителя события и получателей.
- Доставка выполняется через сервис подписок и Kafka/очередь.

---

## 6. Типичные ошибки

1. Сказать, что Издатель сам отправляет сообщение каждому подписчику.
2. Не объяснить ФТ.
3. Сказать, что Подписчик 1/2/3 — обязательно люди.
4. Забыть направление сообщений.
5. Забыть Log/Audit.
