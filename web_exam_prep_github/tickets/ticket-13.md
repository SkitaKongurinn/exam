# Билет №13 — Redis, кэш, сессии, pub/sub, Fias cache-aside

## 0. Как понять этот билет простыми словами

Redis — быстрая база «ключ-значение» в памяти.

```text
Redis = быстрый кэш
```

Он нужен, чтобы не ходить каждый раз в PostgreSQL или внешний Fias API.

---

## 1. Теоретический вопрос 1

**Вопрос:** Какие задачи решает Redis в WEB-приложении? Перечислите сценарии: кэш, сессии, pub/sub.

### Простое объяснение

Redis используют для кэша, сессий, временных токенов, кодов 2FA, rate limiting и pub/sub.

Пример: справочник городов редко меняется, поэтому ответы Fias API можно кэшировать.

### Готовый ответ на 1 вопрос

Redis в WEB-приложении решает задачи быстрого доступа к данным. Чаще всего его используют как кэш для часто запрашиваемых данных, например справочника городов, настроек или профилей пользователей. Это снижает нагрузку на PostgreSQL и внешние API.

Также Redis может хранить пользовательские сессии, временные токены, коды 2FA и данные для ограничения частоты запросов. Ещё один сценарий — Redis pub/sub, когда приложение публикует сообщения, а подписчики получают их в реальном времени.

---

## 2. Теоретический вопрос 2

**Вопрос:** Как Redis используется в crm-cluster? Какие данные целесообразно кэшировать?

### Простое объяснение

В crm-cluster Redis — общий быстрый слой кэша и временных данных.

Кэшировать можно: города, ответы Fias API, настройки, сессии, временные токены, часто читаемые подписки. Платежи кэшировать осторожно.

### Готовый ответ на 2 вопрос

В crm-cluster Redis используется как быстрый инфраструктурный компонент для кэширования и временного хранения данных. Он помогает backend-сервисам быстрее получать часто используемую информацию и снижает нагрузку на PostgreSQL и внешние сервисы.

Целесообразно кэшировать справочники, например города из Fias API, настройки, данные сессий, временные токены, результаты частых запросов и редко изменяемые справочные данные. Для таких данных можно задавать TTL. Платёжные и критичные данные нужно кэшировать осторожно, чтобы не показывать устаревшую информацию.

## 3. Практическое задание

**Задание:** Опишите стратегию кэширования справочника «Города» через Fias API.

### Решение

Ключ:

```text
city:{cityId}
```

TTL: 24 часа или 7 дней.

Алгоритм cache-aside:

```text
1. Backend получает cityId.
2. Проверяет Redis по ключу city:{cityId}.
3. Если данные есть — возвращает их.
4. Если данных нет — обращается к Fias API.
5. Сохраняет ответ в Redis с TTL.
6. Возвращает город клиенту.
```

Псевдокод:

```ts
async function getCityById(cityId: string): Promise<CityDto> {
    const cacheKey = `city:${cityId}`;

    const cached = await redis.get(cacheKey);
    if (cached) {
        return JSON.parse(cached);
    }

    const city = await fiasApi.getCityById(cityId);

    await redis.set(cacheKey, JSON.stringify(city), { ttl: 86400 });

    return city;
}
```

---

## 4. Краткая версия для устного пересказа

Redis — быстрая in-memory база ключ-значение. В WEB-приложении он используется для кэша, сессий, временных токенов и pub/sub. Для Fias API подходит cache-aside: сначала ищем город в Redis, если нет — идём во Fias API, сохраняем с TTL и возвращаем.

---

## 5. Что обязательно сказать

- Redis = быстрый кэш в памяти.
- TTL = время жизни записи.
- Cache-aside = сначала кэш, потом источник.
- Города можно кэшировать.
- Redis снижает нагрузку на БД и API.

---

## 6. Типичные ошибки

1. Сказать, что Redis заменяет PostgreSQL.
2. Не указать TTL.
3. Не описать cache-aside.
4. Кэшировать критичные платежи без обновления.
5. Забыть ключ кэша.
