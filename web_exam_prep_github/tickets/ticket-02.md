## 0. Как понять этот билет простыми словами

```text
User → Subscription → subscriber_type 1/2/3 → status
Сервис подписок → Kafka/RabbitMQ → обработчики
```

## 1. Теоретический вопрос 1

**Вопрос:** Что представляет собой сущность Subscriptions в CRM-системе? Какие поля и связи должна содержать модель подписки?

### Простое объяснение

**Subscriptions** — сущность подписки пользователя. Она отвечает на вопросы: кто подписался, какой тип подписчика выбран, активна ли подписка, когда создана.

Типовые поля: `id`, `user_id`, `subscriber_type`, `status`, `created_at`, `updated_at`, `cancelled_at`.

Связь: один User может иметь много Subscription.

### Готовый ответ на 1 вопрос

Сущность Subscriptions в CRM-системе описывает подписку пользователя на определённые события или сервисы. Она связывает пользователя с типом подписчика и текущим статусом подписки.

Модель подписки должна содержать первичный ключ, внешний ключ на пользователя, тип подписчика 1/2/3, статус подписки, дату создания и дату обновления. Дополнительно можно хранить дату отмены подписки. Через эту сущность система понимает, кто является подписчиком, какие события ему нужно доставлять и активна ли его подписка.

## 2. Теоретический вопрос 2

**Вопрос:** Как сервис подписок взаимодействует с Kafka и очередями сообщений? Объясните, зачем нужны RabbitMQ и Kafka в crm-cluster и чем они отличаются по назначению.

### Простое объяснение

Сервис подписок получает событие и публикует его в брокер сообщений.

```text
Сервис подписок → Kafka/RabbitMQ → подписчики/обработчики
```

Разница:

```text
Kafka = поток событий, история, много потребителей.
RabbitMQ = очередь задач, доставка сообщения обработчику.
```

### Готовый ответ на 2 вопрос

Сервис подписок взаимодействует с Kafka и очередями сообщений для асинхронной доставки событий. Когда появляется событие, например «новая подписка», сервис подписок публикует его в брокер сообщений. После этого подписчики или другие сервисы получают событие и обрабатывают его независимо от основного запроса.

Kafka в crm-cluster используется как потоковая платформа событий: она удобна для хранения истории событий и доставки одного события нескольким потребителям. RabbitMQ чаще используется как очередь задач, где сообщение передаётся обработчику для выполнения конкретной работы. Оба инструмента разгружают backend и повышают отказоустойчивость.

## 3. Практическое задание

**Задание:** Спроектируйте SQL-схему таблицы `subscriptions` в PostgreSQL.

### Решение

```sql
CREATE TABLE subscriptions (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id UUID NOT NULL,
    subscriber_type SMALLINT NOT NULL,
    status VARCHAR(20) NOT NULL DEFAULT 'active',
    created_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP NOT NULL DEFAULT CURRENT_TIMESTAMP,
    cancelled_at TIMESTAMP NULL,

    CONSTRAINT fk_subscriptions_user
        FOREIGN KEY (user_id)
        REFERENCES users(id)
        ON DELETE CASCADE,

    CONSTRAINT chk_subscriber_type
        CHECK (subscriber_type IN (1, 2, 3)),

    CONSTRAINT chk_subscription_status
        CHECK (status IN ('active', 'pending', 'cancelled'))
);
```

## 4. Краткая версия для устного пересказа

Subscriptions — это таблица/сущность подписок. Она связывает пользователя с типом подписчика 1/2/3, статусом и датой создания. Сервис подписок получает события и отправляет их в Kafka или RabbitMQ. Kafka — поток событий и история, RabbitMQ — очередь задач и доставка сообщений обработчикам.

---

## 5. Что обязательно сказать

- Subscriptions связывает User и тип подписчика.
- Статус показывает, активна подписка или отменена.
- Kafka — поток событий.
- RabbitMQ — очередь задач.
- Сервис подписок делает доставку асинхронной.
