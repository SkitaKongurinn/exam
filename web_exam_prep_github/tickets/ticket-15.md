# Билет №15 — Assessment, User, Payment, Logic, Save-1

## 0. Как понять этот билет простыми словами

Билет про бизнес-логику расчёта платежей.

```text
User → Subscription → Payment
Assessment считает сумму
```

---

## 1. Теоретический вопрос 1

**Вопрос:** Что включает модуль Assessment? Опишите связь сущностей User и Payment в контексте оценки/начисления.

### Простое объяснение

Assessment — модуль оценки/начисления. Он рассчитывает платежи по правилам: базовая ставка, месяцы, тип подписчика, коэффициент.

Связь:

```text
User 1 --- * Payment
```

### Готовый ответ на 1 вопрос

Модуль Assessment включает бизнес-логику оценки, начисления или расчёта платежей. Он может учитывать тип подписчика, длительность подписки, базовую ставку и коэффициенты. Результатом работы модуля может быть создание Payment.

User и Payment связаны так: пользователь может иметь один или несколько платежей. Payment хранит сумму, дату, статус и ссылку на пользователя. Assessment определяет сумму для конкретного пользователя и формирует Payment.

---

## 2. Теоретический вопрос 2

**Вопрос:** Что означают узлы Logic и Save-1 на карте? Как бизнес-логика отделена от слоя сохранения данных?

### Простое объяснение

Logic — слой бизнес-правил: рассчитать сумму, проверить тип подписчика, применить коэффициент.

Save-1 — операция сохранения результата: сохранить Payment, записать audit.

```text
Logic считает.
Repository сохраняет.
```

### Готовый ответ на 2 вопрос

Узел Logic означает слой бизнес-логики приложения. В нём находятся правила расчёта, проверки и принятия решений. Например, расчёт платежа по типу подписчика и количеству месяцев относится к бизнес-логике.

Save-1 можно понимать как операцию сохранения результата бизнес-логики. После расчёта платежа система сохраняет Payment в БД. Бизнес-логика должна быть отделена от слоя сохранения: service/logic рассчитывает данные, а repository отвечает за запись в PostgreSQL.

## 3. Практическое задание

**Задание:** Напишите алгоритм расчёта платежа: базовая ставка × месяцы × коэффициент типа подписчика.

### Решение

```ts
type SubscriberType = 1 | 2 | 3;

function calculatePayment(subscriberType: SubscriberType, months: number): number {
    const baseRate = 1000;

    if (months <= 0) {
        throw new Error('Months must be greater than zero');
    }

    const coefficients: Record<SubscriberType, number> = {
        1: 1.0,
        2: 1.5,
        3: 2.0
    };

    return baseRate * months * coefficients[subscriberType];
}
```

Пример:

```text
subscriberType = 2
months = 3
payment = 1000 × 3 × 1.5 = 4500
```

## 4. Краткая версия для устного пересказа

Assessment рассчитывает начисления и платежи. User может иметь много Payment. Logic — бизнес-правила, Save-1 — сохранение результата. Расчёт нужно отделять от Repository: Logic считает, Repository сохраняет.

---

## 5. Что обязательно сказать

- Assessment рассчитывает начисления.
- User может иметь много Payment.
- Logic = бизнес-правила.
- Save-1 = сохранение результата.
- Repository сохраняет, Logic считает.

---

## 6. Типичные ошибки

1. Смешать расчёт и SQL в одной функции.
2. Забыть проверить months > 0.
3. Не объяснить User-Payment.
4. Сказать, что Assessment — это только таблица.
5. Не отделить Logic от Save-1.
