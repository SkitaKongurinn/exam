# Билет №17 — идентификация, аутентификация, авторизация, password, 2FA, биометрия

## 0. Как понять этот билет простыми словами

Безопасный вход состоит из трёх этапов:

```text
Идентификация → Аутентификация → Авторизация
```

Кто ты? Докажи. Что тебе можно?

---

## 1. Теоретический вопрос 1

**Вопрос:** Дайте определения: идентификация, аутентификация, авторизация. В каком порядке они выполняются при входе?

### Простое объяснение

Идентификация — пользователь сообщает, кто он: email/login.

Аутентификация — доказывает, что это он: password/2FA.

Авторизация — система проверяет права: читать, создавать, удалять.

Порядок: идентификация → аутентификация → авторизация.

### Готовый ответ на 1 вопрос

Идентификация — процесс определения пользователя, ответ на вопрос «кто ты?». Обычно пользователь вводит email, логин или телефон.

Аутентификация — проверка подлинности пользователя, доказательство, что он действительно является тем, за кого себя выдаёт. Используются пароль, одноразовый код, 2FA или другие методы. Авторизация — проверка прав доступа: какие действия пользователь может выполнять после входа. Порядок: сначала идентификация, затем аутентификация, затем авторизация.

---

## 2. Теоретический вопрос 2

**Вопрос:** Опишите методы аутентификации: password, 2FA, биометрия. Какие применимы в WEB-приложении?

### Простое объяснение

Password — проверка пароля, в БД хранится hash.

2FA — второй фактор: код из приложения, SMS/email, push.

Биометрия — лицо, голос, отпечаток и т.д. В WEB чаще используют password + 2FA, биометрия сложнее.

### Готовый ответ на 2 вопрос

Password-аутентификация основана на проверке пароля пользователя. В базе должен храниться не сам пароль, а его хэш. 2FA добавляет второй фактор подтверждения: код из приложения, SMS, email или push-уведомление. Это повышает безопасность, потому что одного пароля недостаточно.

Биометрия использует физические признаки пользователя: лицо, голос, отпечаток пальца. В WEB-приложении наиболее применимы password и 2FA. Биометрия возможна через устройства и специальные API, но сложнее в реализации и требует внимания к безопасности персональных данных.

## 3. Практическое задание

**Задание:** Опишите flow входа пользователя: email + password + 2FA → JWT/session token.

### Решение

```text
1. Пользователь открывает форму входа.
2. Вводит email и password.
3. Web client отправляет данные на backend.
4. Backend ищет пользователя по email.
5. Backend проверяет пароль через hash compare.
6. Если пароль неверный — ошибка.
7. Если пароль верный — создаётся 2FA challenge.
8. Пользователь вводит 2FA-код.
9. Web client отправляет 2FA-код на backend.
10. Backend проверяет код.
11. Если код верный — backend создаёт JWT/session token.
12. Backend возвращает token клиенту.
13. Клиент использует token в следующих запросах.
14. Backend проверяет token и права пользователя.
```

Схема:

```text
User → Web client: email + password
Web client → Backend: login request
Backend → DB: find user
Backend → Backend: check password hash
Backend → User: request 2FA code
User → Web client: 2FA code
Web client → Backend: verify 2FA
Backend → Web client: JWT/session token
```

---

## 4. Краткая версия для устного пересказа

Идентификация — кто пользователь, аутентификация — доказал ли он это, авторизация — что ему разрешено. Сначала email, потом пароль, потом 2FA, затем backend выдаёт JWT/session token. В WEB обычно применимы password и 2FA, биометрия возможна, но сложнее.

---

## 5. Что обязательно сказать

- Идентификация = кто ты.
- Аутентификация = докажи.
- Авторизация = что можно.
- Пароль хранится как hash.
- 2FA повышает безопасность.

---

## 6. Типичные ошибки

1. Перепутать authentication и authorization.
2. Хранить пароль в открытом виде.
3. Забыть 2FA.
4. Выдать token до 2FA.
5. Не проверять права после входа.
