# Билет №16 — Log, Audit, audit trail, JSON записи аудита

## 0. Как понять этот билет простыми словами

Log и Audit — оба журналы, но для разного.

```text
Log = техническая диагностика
Audit = кто, когда и что изменил
```

---

## 1. Теоретический вопрос 1

**Вопрос:** В чём разница между Log и Audit? Какие события записываются в каждый из журналов?

### Простое объяснение

Log: ошибки, предупреждения, время запроса, падение Kafka, ошибка БД.

Audit: кто создал платёж, кто изменил подписку, старое/новое значение, время и IP.

### Готовый ответ на 1 вопрос

Log и Audit отличаются назначением. Log — технический журнал для диагностики и отладки. В него записываются ошибки, предупреждения, время выполнения запросов, проблемы подключения к базе или брокеру сообщений.

Audit — журнал значимых действий пользователей и системы. Он нужен, чтобы понять, кто, когда и что изменил. В audit записываются операции создания, изменения и удаления важных сущностей, особенно платежей, пользователей и подписок. Audit должен содержать пользователя, действие, сущность, старое и новое значение.

---

## 2. Теоретический вопрос 2

**Вопрос:** Какие требования предъявляются к audit trail в WEB-приложении с платежами? Что фиксировать при Save-1?

### Простое объяснение

Audit trail — цепочка записей, по которой можно восстановить историю действий.

При Save-1 фиксируют: timestamp, userId, action, entityType, entityId, oldValue, newValue, ipAddress, result.

### Готовый ответ на 2 вопрос

В WEB-приложении с платежами audit trail должен быть полным, достоверным и защищённым от незаметного изменения. Он должен позволять восстановить историю операций: кто создал или изменил платёж, когда это произошло, с какого IP-адреса и какие данные были изменены.

При операции Save-1 обязательно фиксируются timestamp, userId, действие, тип сущности, id сущности, старое значение, новое значение, IP-адрес и результат операции. Для платежей это особенно важно, потому что такие действия имеют финансовое значение.

## 3. Практическое задание

**Задание:** Опишите JSON-формат записи аудита для операции «создание платежа».

### Решение

```json
{
  "timestamp": "2026-06-09T12:30:00Z",
  "userId": "user-123",

  "newValue": {

    "status": "created",
    "paymentDate": "2026-06-09T12:30:00Z"
  },

  "result": "success"
}
```

## 4. Краткая версия для устного пересказа

Log — технический журнал для ошибок и диагностики. Audit — журнал значимых действий: кто, когда и что изменил. В платежах audit trail должен фиксировать пользователя, действие, сущность, старое и новое значение, время, IP и результат. Save-1 должен создавать audit-запись.

---

## 5. Что обязательно сказать

- Log = технические события.
- Audit = история действий.
- Audit важен для платежей.
- oldValue/newValue показывают изменение.
- Save-1 фиксируется в audit.

---

## 6. Типичные ошибки

1. Считать Log и Audit одним и тем же.
2. Не указать oldValue/newValue.
3. Не указать userId.
4. Не фиксировать IP.
5. Логировать платёж только как техническое сообщение.
